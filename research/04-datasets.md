# Datasets & Data Curation for Useful Tiny Models

**TL;DR:** The data you train on is more important than the architecture you choose. The landmark TinyStories result showed that models below 10M parameters become coherent *not* because of architectural magic, but because the dataset was simple, consistent, and high-quality enough for the model to extract signal from. Modern best practice — canonized by HuggingFace's SmolLM2 and Microsoft's phi series — is to curate toward *educational density*: every token should teach something. On 8 GB VRAM, the right dataset strategy is to start tiny (TinyStories, ~2 GB), iterate fast, then step up to FineWeb-Edu 10B-token sample (~27 GB) once your architecture and training loop are solid.

---

## 1. Why Data Quality Dominates at Small Scale

When model capacity is small, every gradient update competes with noise. A model with 10M parameters has roughly 10M "slots" to compress the world into. If the training corpus is full of contradictory facts, broken HTML, boilerplate, or malformed sentences, those slots fill with garbage.

The intuition from information theory: a smaller model can only learn from signal that is *already* highly compressed and consistent. Large models can average across noise because they have capacity to memorize many exceptions. Tiny models cannot.

This is why two datasets with the same token count can differ enormously in what a tiny model learns from them: 1B tokens of raw CommonCrawl vs. 1B tokens of textbook-quality synthetic text are not remotely equivalent for a 50M-parameter model.

### The Three Eras of Dataset Thinking

| Era | Representative Dataset | Philosophy |
|---|---|---|
| Pre-2023 | The Pile, C4 | More tokens = better |
| 2023 | TinyStories, phi-1 | Quality over quantity; topic compression |
| 2024-2026 | FineWeb-Edu, SmolLM-Corpus | Classifier-filtered educational density at web scale |

## 2. The Foundational Result: TinyStories

**Paper:** [TinyStories: How Small Can Language Models Be and Still Speak Coherent English?](https://arxiv.org/abs/2305.07759) (Eldan & Li, Microsoft Research, May 2023)

**Dataset:** [roneneldan/TinyStories](https://huggingface.co/datasets/roneneldan/TinyStories)

### What the paper showed

The authors asked: does coherent text generation require hundreds of millions of parameters, or is that an artifact of training on messy web data? They created a synthetic dataset of short stories using only the vocabulary a 3-4 year old knows (around 1,500 common words), generated by GPT-3.5 and GPT-4.

Key quantitative results:
- Models below **10M parameters** trained on TinyStories generated coherent, multi-paragraph stories with near-perfect grammar and narrative consistency
- A model with a **single transformer block** (~1M params) still produced fluent text
- Even a **1M parameter** model ([TinyStories-1M](https://huggingface.co/roneneldan/TinyStories-1M)) told recognizable stories
- The same model architecture trained on OpenWebText (standard web data) at 10M params produced incoherent output

### Why it works: the compression argument

TinyStories is a **closed domain** with a small vocabulary and simple, repetitive syntactic structures. There are perhaps 50 archetypal "story shapes" (character encounters problem → tries to solve it → learns lesson). A 10M model has enough capacity to compress that distribution. The same 10M model cannot compress the full complexity of web text.

This is the core intuition: **match your dataset complexity to your model's capacity budget.**

### Dataset stats

| File | Size | Notes |
|---|---|---|
| TinyStories-train.txt | 1.92 GB | ~2.1B characters |
| TinyStories-valid.txt | 19.4 MB | Validation split |
| TinyStoriesV2-GPT4-train.txt | 2.23 GB | Higher quality (GPT-4 generated) |
| TinyStoriesInstruct train | ~same | Each story preceded by instructions |

**Total download: ~7.6 GB** — fits easily on any machine.

**Token count estimate:** ~475M tokens (at ~0.25 chars/token for simple English text), or use the tokenized version available in [Karpathy's llama2.c](https://github.com/karpathy/llama2.c/blob/master/tinystories.py) which preprocesses this for training.

### Throughput estimate on RTX 3060 Ti

A 163M GPT-2 model on an RTX 3090 (24 GB VRAM) achieves ~20,000 tokens/sec ([source](https://www.gilesthomas.com/2025/12/llm-from-scratch-28-training-a-base-model-from-scratch)). Your RTX 3060 Ti (8 GB, lower memory bandwidth) will be ~40-60% of that:
- Estimated: **8,000-12,000 tokens/sec** for a 50-150M model in BF16/FP16
- At 10,000 tok/s: training on all 475M TinyStories tokens = **~13 hours**
- For a 15M model (much smaller): closer to **~1-2 hours**

This fits the ETH Zurich philosophy: run experiments in minutes to hours, not days.

### The Instruct variant

[roneneldan/TinyStoriesInstruct](https://huggingface.co/datasets/roneneldan/TinyStoriesInstruct) includes structured instructions before each story (word lists, required sentences, features like "bad ending" or "moral value", and a summary). This is an excellent starting point for mini-instruct training: the task is simple enough that even a 10M model can follow it.

## 3. FineWeb and FineWeb-Edu: Web Scale Done Right

**Dataset:** [HuggingFaceFW/fineweb](https://huggingface.co/datasets/HuggingFaceFW/fineweb)
**Dataset:** [HuggingFaceFW/fineweb-edu](https://huggingface.co/datasets/HuggingFaceFW/fineweb-edu)
**Paper:** [The FineWeb Datasets: Decanting the Web for the Finest Text Data at Scale](https://arxiv.org/abs/2406.17557) (NeurIPS 2024 Datasets & Benchmarks Track)

### What it is

FineWeb is a 15-trillion-token dataset derived from 96 CommonCrawl snapshots (2013-2025). It was built with [datatrove](https://github.com/huggingface/datatrove), HuggingFace's modular data-curation library. The pipeline includes:
1. URL filtering (domain blocklists)
2. Trafilatura HTML → text extraction
3. Language detection (keep English)
4. Heuristic quality filters (C4, Gopher rules): min/max line length, stop-word density, repetition ratios
5. MinHash deduplication at document level

FineWeb-Edu went further: they had **Llama-3-70B-Instruct** score 460,000 randomly sampled webpages for educational quality on a 0-5 scale, then trained a **linear regression classifier** on top of [Snowflake-arctic-embed-m](https://huggingface.co/Snowflake/snowflake-arctic-embed-m) embeddings to score the entire FineWeb corpus. Pages scoring ≥ 3 were retained.

The FineWeb-Edu classifier achieved **F1=82%** on the validation set. The resulting dataset contains **1.3T tokens** and significantly outperforms FineWeb on MMLU, ARC, and OpenBookQA.

### Available sample sizes

| Subset | Tokens | Approx disk | Use case |
|---|---|---|---|
| `sample-10BT` | 10B | ~27 GB | Fast iteration, proof of concept |
| `sample-100BT` | 100B | ~270 GB | Serious pretraining run |
| `sample-350BT` | 350B | ~950 GB | Full scale (exceeds your disk) |
| Full dataset | 1.3T | ~3.5 TB | Cluster-scale only |

**For your 660 GB disk:** the `sample-100BT` subset at ~270 GB fits with room to spare. For first experiments, `sample-10BT` at ~27 GB is the right starting point.

```python
from datasets import load_dataset

# Streaming mode — no full download needed
ds = load_dataset("HuggingFaceFW/fineweb-edu", "sample-10BT", 
                  split="train", streaming=True)

# Or download locally for fast random access during training
ds = load_dataset("HuggingFaceFW/fineweb-edu", "sample-10BT", 
                  split="train")
```

### FineWeb vs FineWeb-Edu for tiny models

For models below 150M params, **FineWeb-Edu is strictly better** than FineWeb. The educational filter removes approximately 85% of the web by token count, but what remains is exactly the kind of dense, pedagogically-structured text that tiny models can learn from. Think: Wikipedia articles, textbook excerpts, technical tutorials, educational blog posts.

## 4. SmolLM-Corpus: The Current Best Practice for Tiny Models

**Dataset:** [HuggingFaceTB/smollm-corpus](https://huggingface.co/datasets/HuggingFaceTB/smollm-corpus)
**Paper:** [SmolLM2: When Smol Goes Big — Data-Centric Training of a Small Language Model](https://arxiv.org/abs/2502.02737) (2025)

HuggingFace's SmolLM2 (135M, 360M, 1.7B) is currently the state-of-the-art reference for tiny model training. The 135M and 360M versions were trained on **600B tokens** from the SmolLM-Corpus; the 1.7B on **1T tokens**.

The corpus has three components, each targeting a different capability:

### 4.1 Cosmopedia v2 — Synthetic Textbooks

| Metric | Value |
|---|---|
| Documents | 39.1 million |
| Tokens | 28B |
| Generator | Mixtral-8x7B-Instruct-v0.1 |
| Disk size | ~122 GB |
| Topics | 34,000+ |

Cosmopedia v2 generates textbooks, blog posts, stories, and WikiHow-style articles across 34,000+ topics. The key innovation vs. scraping: every document is *coherent from start to finish* because it was generated with a coherence constraint. The audience distribution: 40% middle school, 30% college, 30% other — designed to cover a range of complexity.

```python
ds = load_dataset("HuggingFaceTB/smollm-corpus", "cosmopedia-v2", 
                  split="train", num_proc=16)
```

**Disk warning:** 122 GB. If disk is tight, stream it. At 28B tokens, it's the largest single chunk of textbook-quality synthetic data publicly available.

### 4.2 Python-Edu — Filtered Educational Code

| Metric | Value |
|---|---|
| Documents | 7.68 million |
| Tokens | ~4B |
| Source | The Stack v2 |
| Filter | Educational quality score ≥ 4 |

Python files from The Stack v2 scored by the same kind of educational classifier used for FineWeb-Edu. Only files scoring 4+ were kept. This dramatically reduces noise (commented-out code, auto-generated boilerplate, cryptic one-liners) and keeps tutorials, learning-focused scripts, and well-commented implementations.

```python
ds = load_dataset("HuggingFaceTB/smollm-corpus", "python-edu", 
                  split="train", num_proc=16)
```

**Size: ~10-15 GB** — very manageable.

### 4.3 FineWeb-Edu (Deduplicated)

| Metric | Value |
|---|---|
| Documents | 190 million |
| Tokens | 220B |
| Disk size | ~600 GB |
| Note | Further deduplicated vs. the main FineWeb-Edu |

This is the web text backbone. For your 660 GB disk, you cannot download this in full while keeping the other components. **Strategy:** use the `sample-10BT` or `sample-100BT` versions from the main FineWeb-Edu dataset, or stream from this one.

### SmolLM2 data mixture (approximate)

For the 135M model (600B training tokens):
- FineWeb-Edu-dedup: ~73% of tokens
- Cosmopedia v2: ~5%
- Python-Edu: ~1%
- Additional math/code/instruct data: ~21%

The lesson: **web text dominates by volume, but synthetic/educational data punches far above its weight.** Even 5% Cosmopedia dramatically improves reasoning and instruction following.

## 5. The phi Line: Textbooks Are All You Need

**Paper:** [Textbooks Are All You Need](https://arxiv.org/abs/2306.11644) (phi-1, Gunasekar et al., Microsoft, June 2023)
**Paper:** [Textbooks Are All You Need II: phi-1.5](https://arxiv.org/abs/2309.05463) (September 2023)

### phi-1: Code quality explosion

phi-1 is a 1.3B transformer trained on:
- ~6B tokens of "textbook quality" web data (filtered using GPT-4 quality annotations)
- ~1B tokens of GPT-3.5-generated synthetic Python textbooks and exercises

Result: **50.6% pass@1 on HumanEval** (Python coding benchmark), comparable to models 10x its size. phi-1-small at 350M parameters achieved 45% — with the same pipeline and far less compute.

### phi-1.5: Generalization beyond code

phi-1.5 is 1.3B parameters trained on 30B tokens: phi-1's data (7B) plus ~20B tokens of newly synthesized "textbook-like" data covering common sense reasoning, factual knowledge, and logical reasoning.

Results: near-SOTA on common sense benchmarks among models <10B parameters, despite training on almost entirely synthetic data.

### The core insight for your project

phi demonstrates that **1B synthetic textbook-quality tokens > 100B raw web tokens** for a small model. The practical implication: if you're building a domain-specific tiny model (e.g., a coding assistant or a math helper), generate 1-5B tokens of high-quality synthetic domain data using a capable API model (GPT-4o, Claude, or a local Mixtral). This is cheaper and faster than curating billions of web tokens.

The [distilabel](https://github.com/argilla-io/distilabel) framework automates this generation pipeline and was used to build SmolTalk (see Section 7).

## 8. Data Curation Pipeline: datatrove

**Repo:** [huggingface/datatrove](https://github.com/huggingface/datatrove)

datatrove is HuggingFace's open-source data processing library, used to build FineWeb and FineWeb-Edu. If you want to create your own curated dataset (e.g., from a domain-specific crawl or a subset of CommonCrawl), this is the tool.

Key pipeline stages you'll encounter:

### 8.1 Heuristic Quality Filters

Borrowed from the C4 and Gopher papers:
- **Document-level:** min/max word count, fraction of words with alphabetic characters, stop-word density
- **Line-level:** remove lines starting with `<`, lines with high symbol ratios, lines shorter than 100 chars (often navigation/UI text)
- **Repetition:** if the top 5-gram appears in >10% of the document, reject (boilerplate detection)

### 8.2 MinHash Deduplication

Near-duplicate detection using MinHash LSH (Locality Sensitive Hashing). Key parameters:
- **Shingle size:** 5-grams (character or word level)
- **Number of hash functions:** 112 (14 bands × 8 rows in FineWeb)
- **Similarity threshold:** ~0.75 (documents 75%+ similar are duplicates)

Deduplication matters enormously: CommonCrawl has ~30-40% near-duplicate content. Training on duplicates wastes capacity and hurts generalization.

```bash
# Install datatrove
pip install datatrove[processing]

# Run MinHash dedup (from their examples)
python examples/minhash_deduplication.py
```

### 8.3 Model-Based Quality Filtering

The state of the art (DCLM, FineWeb-Edu) trains a lightweight classifier on LLM quality annotations:
1. Sample ~500K documents
2. Have an LLM (Llama-3-70B, GPT-4o) score each document 0-5 on your quality criterion
3. Train a small embedding model + linear head on these annotations
4. Run inference over the full corpus in hours (much cheaper than LLM scoring everything)

**[DCLM](https://arxiv.org/abs/2406.11794)** (DataComp for Language Models, NeurIPS 2024) showed that model-based filtering was the primary axis controlling final performance, enabling a 7B model trained on 2T tokens to reach 63% 5-shot MMLU — 7x less compute than Llama-3 8B.

## 9. Data-to-Parameter Ratios: How Much Data Do You Need?

### Chinchilla baseline (2022)

The [Chinchilla paper](https://arxiv.org/abs/2203.15556) established: for compute-optimal training, use **~20 tokens per parameter**. For a 50M model: ~1B tokens.

### Modern practice: massively over-train small models

In 2024-2026, the community has shifted dramatically. Inference cost matters more than training compute — and a smaller model costs less per query. So: **train longer on more data** to make the model as capable as possible at its size.

| Model | Params | Tokens | Tokens/Param ratio |
|---|---|---|---|
| Chinchilla-optimal | any | 20× params | 20 |
| Llama-2-7B | 7B | 2T | ~286 |
| Llama-3-8B | 8B | 15T | ~1,875 |
| SmolLM2-135M | 135M | 600B | ~4,444 |
| SmolLM2-360M | 360M | 600B | ~1,667 |
| Qwen3-0.6B | 0.6B | 36T | ~60,000 |

**The implication for your 50M model:** Chinchilla says 1B tokens, but a well-trained tiny model would benefit from 50-200B tokens if you have the time. The practical constraint is training time:
- At ~10,000 tok/s on RTX 3060 Ti:
  - 1B tokens ≈ ~28 hours
  - 10B tokens ≈ ~11.6 days
  - 50B tokens ≈ ~58 days (impractical for fast iteration)

**For the ETH Zurich fast-iteration approach:** train on 100M-500M tokens initially (6 hours to 14 hours). This is enough to validate that the model can learn at all. Once architecture is stable, run a 1-3B token training run as your "final" experiment.

### Quality amplifies the ratio

If your data is high-quality (TinyStories, FineWeb-Edu), **you need fewer tokens** to reach the same quality level. A rough heuristic: educational-quality data is worth ~5-10x the same number of raw web tokens. A 50M model trained on 200M tokens of TinyStories may outperform the same model trained on 1B tokens of raw web text.

## 12. Tokenization Considerations

The tokenizer is part of your data pipeline and affects what "tokens" actually means.

| Tokenizer | Vocab size | Notes |
|---|---|---|
| GPT-2 BPE | 50,257 | Default for TinyStories/nanoGPT experiments |
| Llama-2/3 SentencePiece | 32,000 / 128,000 | Good general purpose |
| SmolLM2 (same as Llama-2) | 49,152 | Matches SmolLM-Corpus token counts |
| Custom BPE | 4,096-16,000 | For very tiny models to reduce embedding table size |

For models below 30M parameters, vocabulary size matters significantly for VRAM: a 50K-vocab embedding table at float32 is 50K × 128 dims × 4 bytes = 25 MB — significant at this scale. Consider a custom BPE tokenizer with 8K-16K tokens trained on your specific domain.

[SentencePiece](https://github.com/google/sentencepiece) or HuggingFace's [tokenizers](https://github.com/huggingface/tokenizers) (Rust-based, fast) can train a custom tokenizer on your corpus in minutes.

## 13. Learn-by-Doing: Experiments

These are ordered from fastest-to-run to most informative for dataset understanding.

### Experiment 1: The Coherence Threshold on TinyStories (45 min)

Train four models of increasing size — 1M, 3M, 10M, 28M — on TinyStories for 50K steps with a small batch. Use Karpathy's [llama2.c](https://github.com/karpathy/llama2.c) or [nanoGPT](https://github.com/karpathy/nanoGPT). Generate 5 stories from each model. Answer: at what parameter count does the output become clearly coherent? Does it match the paper's claim of <10M?

**You'll learn:** Firsthand intuition for the coherence threshold. The gap between 3M and 10M should be striking.

### Experiment 2: Dataset Quality vs. Quantity (2-3 hours)

Train two 15M-parameter models for the same number of steps (~50K):
- Model A: OpenWebText (or a raw CommonCrawl slice)
- Model B: FineWeb-Edu 10BT sample

Log validation loss every 1K steps. At the end, generate 10 text samples from each. Compare both the loss curves and the output quality.

**You'll learn:** Viscerally why "quality over quantity" is real and quantifiable. You'll see the loss curves diverge early and the outputs differ dramatically.

### Experiment 3: Domain Transfer — Does TinyStories Generalize? (1-2 hours)

Take your trained TinyStories 15M model and evaluate it on:
1. Short Wikipedia sentences (perplexity)
2. Simple Python docstrings (perplexity)
3. A custom prompt about a topic not in TinyStories (e.g., "explain how photosynthesis works")

Measure: how much does the model's coherence on out-of-domain text degrade vs. in-domain stories?

**You'll learn:** The cost of a closed-domain training set. This will motivate your decision about when to graduate from TinyStories to a broader corpus.

### Experiment 4: Mixing Ratios (3-4 hours)

Train three 30M-parameter models on 500M tokens each, varying the data mixture:
- Mix A: 100% FineWeb-Edu
- Mix B: 80% FineWeb-Edu + 20% Cosmopedia v2
- Mix C: 60% FineWeb-Edu + 30% Cosmopedia v2 + 10% Python-Edu

Evaluate on: general text perplexity, a simple factual Q&A prompt set (10 questions), and a Python code completion task (10 functions). Track which mix gives the best all-around performance.

**You'll learn:** How data mixing ratios affect capability profiles, and why the SmolLM2 team landed on the mixture they did. You'll develop intuition for the trade-offs that took HuggingFace months of experimentation.

```
Step 4: Instruct-tune
  → SmolTalk (100K-200K samples), 1-3 epochs, ~4-8 hours

Code path:
  → Python-Edu (10-15 GB) as primary pretraining, same ladder
```

The single most important insight: **start with TinyStories.** It is the fastest way to get a working, coherent language model, and everything you learn there transfers directly to the larger-scale experiments.
